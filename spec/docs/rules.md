# Rule Authoring Guide

Rules are the heart of tripwire. They encode the patterns that distinguish a suspicious filesystem event from a normal one. This guide explains the rule format, how match expressions work, and how to add a new rule end-to-end.

In v0.2, tripwire is a **runtime detection daemon**. Rules evaluate against filesystem events (`read`, `write`, `open`, etc.) generated by `fanotify` / `fsevents`, tagged with the firing process's identity and agent-ancestry classification. Rules that scanned install-script content (the v0.1 `pattern.*` family) are removed; see `docs/history.md` if you need that prior context.

## File layout

Rules live as YAML files under `packages/rules/patterns/`. One file per category; multiple rules per file are fine.

```
packages/rules/
├── patterns/
│   ├── credential-paths.yaml
│   ├── agent-config-tampering.yaml
│   ├── persistence.yaml
│   └── shell-rc.yaml
├── default-allowlist.yaml
└── iocs/
    └── packages.json
```

Schema: `schemas/rule.schema.json`. CI validates every rule file on PR.

## Rule anatomy

A rule is a YAML document with this shape:

```yaml
id: cred.aws-credentials-read
name: "AWS credentials file read"
severity: high
category: credential-access
description: |
  A process read ~/.aws/credentials or ~/.aws/config. Expected only from
  the AWS CLI directly invoked by a human shell; alerts when an agent
  subprocess or package-manager subprocess does it.
applies_to:
  event_kind: [read, open]
  path:
    home_relative: [".aws/credentials", ".aws/config"]
  ancestry_category:
    not_in: [human-shell]
references:
  - https://example.com/aws-cred-exfil-writeup
authored: 2026-05-16
```

### Field reference

| Field | Required | Description |
|-------|----------|-------------|
| `id` | yes | Dotted lowercase. Format: `<type>.<family>.<specific>`. Types: `ioc`, `cred`, `persist`, `net`, `meta`. Must be globally unique. |
| `name` | yes | Human-readable. Shown in terminal output and the TUI. |
| `severity` | yes | One of: `critical`, `high`, `medium`, `low`, `info`. See severity guide below. |
| `category` | yes | One of: `credential-access`, `persistence`, `defense-evasion`, `exfiltration`, `metadata`. |
| `description` | yes | What this catches, in plain language. |
| `applies_to` | yes | Which events this rule evaluates against. See below. |
| `references` | recommended | Public URLs that justify the rule (writeups, CVEs, advisories). |
| `authored` | recommended | ISO date. |
| `tags` | optional | Free-form tags. |
| `disabled` | optional | If `true`, rule is loaded but never fires. Useful for staging. |

## Severity levels

| Level | When to use | Default action |
|-------|-------------|----------------|
| `critical` | Confirmed IoC match (responsible package on the IoC list) **plus** a credential read, or other unambiguous credential-theft pattern. Almost zero false positives expected. | Notification fires regardless of snooze (`critical` bypasses snooze; documented in spec). |
| `high` | Strong indicator with a small false-positive surface. Patterns like `agent-subprocess` reads `~/.aws/credentials`, or write to `.claude/settings.json` from non-editor ancestry. | Notification fires unless snoozed. |
| `medium` | Suspicious behavior with legitimate uses (e.g. an unknown-ancestry process reads `~/.npmrc`). | Notification fires unless snoozed; logged for investigation. |
| `low` | Heuristic on its own (a write to a shell rc file from any ancestry). | Logged; surfaced in `tripwire status` / the TUI. Not notified by default. |
| `info` | Informational; no security implication on its own but useful in correlation. | Logged only. |

When unsure, start at `medium` and review false-positive rate on the corpus before promoting.

## `applies_to`

The runtime rule engine evaluates each rule against every incoming filesystem event. `applies_to` is the predicate set; all sub-fields are ANDed together.

```yaml
applies_to:
  event_kind: [read, open, write, create, unlink, rename]
  path:
    home_relative:   [".ssh/id_rsa", ".aws/credentials"]   # ~ expansion
    glob:            ["**/.claude/settings.json"]          # glob pattern
    starts_with:     ["~/.config/gh/"]
  ancestry_category:
    in:     [agent-subprocess, package-manager-spawned]   # only fire when category in list
    not_in: [human-shell, agent-direct]                   # never fire when category in list
  ecosystem:
    in: [npm, pypi]                                       # filter by ecosystem of responsible package, if known
```

### `event_kind`

The filesystem event kind. Cross-platform normalized:

| Kind | Linux fanotify | macOS fsevents |
|------|----------------|----------------|
| `read` | `FAN_ACCESS` | (best-effort, see spec §6.3) |
| `open` | `FAN_OPEN` | (best-effort) |
| `write` | `FAN_MODIFY` | `FSE_CONTENT_MODIFIED` |
| `create` | `FAN_CREATE` | `FSE_CREATE_FILE` |
| `unlink` | `FAN_DELETE` | `FSE_DELETE` |
| `rename` | `FAN_MOVED_FROM`/`TO` | `FSE_RENAME` |

### `path`

Match the filesystem path the event touched. At least one of:

- `home_relative` — array of paths relative to `$HOME`. Tilde expansion handled automatically.
- `glob` — array of glob patterns (`**`, `*`, `?`, character classes).
- `starts_with` — array of literal prefixes (after tilde expansion).
- `equals` — array of exact-match absolute paths (rare).

### `ancestry_category`

**The primary noise-control axis.** Limits the rule to events from specific ancestry classifications:

| Category | Meaning |
|----------|---------|
| `human-shell` | Interactive shell session (you typing in a terminal). |
| `agent-direct` | The firing process *is* a known agent binary (claude-code, cursor, etc.). |
| `agent-subprocess` | Any ancestor is a known agent or has an agent identity env var. |
| `package-manager-direct` | The firing process is a known package manager. |
| `package-manager-spawned` | Any ancestor is a known package manager, firing process is not (postinstall, build script, etc.). |
| `unknown` | Nothing matched. |

Use `in:` to whitelist categories, `not_in:` to blacklist. Most credential rules use `not_in: [human-shell]` — humans using `ssh`/`aws`/`gh` directly are allowlisted, but the same path read from any other ancestry fires.

### `ecosystem`

When the responsible installed package can be identified (the firing executable is inside `node_modules` or `site-packages` belonging to a tracked package), the engine attaches package metadata. Rules can scope by it: `ecosystem: { in: [npm] }` means "only fire when the responsible package is an npm package."

This is best-effort — many events come from binaries outside any tracked package (system tools, user scripts). Rules that omit `ecosystem` fire regardless.

## Match logic

The MVP rule engine **does not need** an expression language for matching path content (we don't read file content during evaluation — we react to access events). The rule body is just `applies_to`; the engine evaluates predicates, no `match: any_of` combinator is required.

If a rule needs more sophisticated logic — e.g. "fire if path read AND a network connection to a non-allowlisted host within 30 s" — that's correlation logic, handled by the Phase 2 `net-correlator` package, not by the rule engine.

## Default allowlist

Each rule's behavior is shaped by `packages/rules/default-allowlist.yaml`. Allowlist entries are checked **before** rule evaluation; matching entries suppress the rule entirely.

```yaml
- rule: cred.aws-credentials-read
  process_path: "**/bin/aws"
  ancestry_category: human-shell
  path:
    home_relative: [".aws/credentials", ".aws/config"]
  reason: "AWS CLI direct invocation"
```

The default allowlist is the single most important contribution surface for the project — see CONTRIBUTING.md.

## Writing a rule end-to-end

Let's walk through adding a rule for a hypothetical campaign: malicious npm packages write `~/.config/claude/secrets.json` to stage exfiltration data for the next Claude Code session to read.

### Step 1 — Write the rule

`packages/rules/patterns/agent-config-tampering.yaml`:

```yaml
- id: persist.claude-secrets-drop
  name: "Drop of staged secrets file under ~/.config/claude/"
  severity: high
  category: persistence
  description: |
    A non-editor process wrote to ~/.config/claude/secrets.json. Claude
    Code does not use that path for user data; a write here is a likely
    persistence / staging payload for the next agent session.
  applies_to:
    event_kind: [write, create]
    path:
      home_relative: [".config/claude/secrets.json"]
    ancestry_category:
      not_in: [agent-direct]   # claude-code itself never writes there
  references:
    - https://example.com/claude-config-tampering-writeup
  authored: 2026-05-16
```

### Step 2 — Create a positive fixture

`test/fixtures/rules/persist.claude-secrets-drop/positive/`:

```
ancestry.yaml          # synthetic process tree
event.yaml             # the fs event we expect to fire
```

`ancestry.yaml`:

```yaml
chain:
  - exe: /sbin/init
    argv: [init]
  - exe: /usr/lib/systemd/systemd
    argv: [systemd, "--user"]
  - exe: /bin/bash
    argv: [bash, -c, "node ./postinstall.js"]
  - exe: /usr/local/bin/node
    argv: [node, "./postinstall.js"]
env_markers: {}
expected_category: package-manager-spawned
```

`event.yaml`:

```yaml
kind: write
path: ~/.config/claude/secrets.json
expected_fire: true
```

### Step 3 — Create a negative fixture

`test/fixtures/rules/persist.claude-secrets-drop/negative/`:

```yaml
# ancestry: claude-code itself
chain:
  - exe: /Applications/Claude.app/Contents/MacOS/claude-code
    argv: [claude-code]
env_markers:
  CLAUDE_CODE_SESSION: "fixture-session"
expected_category: agent-direct
```

```yaml
kind: write
path: ~/.config/claude/secrets.json
expected_fire: false   # agent-direct ancestry; rule excludes
```

### Step 4 — Run the rule test harness

```bash
pnpm test:rules --filter persist.claude-secrets-drop
```

Asserts every positive fires and every negative doesn't.

### Step 5 — Check false-positive rate

```bash
pnpm test:false-positive-corpus --rule persist.claude-secrets-drop
```

Scans the curated top-100 workflows corpus (`test/fixtures/allowlist-corpus/`). Any unexpected fires get reported. If false-positive rate is non-zero, tighten the rule before merging.

### Step 6 — PR

Include in your PR:

- The rule file change.
- The fixture directories.
- A note in the PR description: what attack pattern this catches, why now, link to writeup.

## Tuning false positives

The most common reason a new rule misbehaves: it's too broad on `ancestry_category`. The recipe:

1. **Start with `not_in: [human-shell]`** for any credential read. Humans using the canonical tool to read their own credentials are not the attack.
2. **Add `not_in: [agent-direct]`** if the agent itself legitimately uses the path (rare for credentials, common for settings files).
3. **Use `in: [package-manager-spawned, unknown]`** for very high-signal rules where any other ancestry is unambiguously suspicious.
4. **Allowlist before tightening.** If a specific tool reads the path benignly, add it to `default-allowlist.yaml` rather than narrowing the rule — the rule still catches everything else.
5. **Test against the corpus.** The corpus is a curated set of normal workflows (`ssh`, `aws sso login`, `gh auth login`, `npm install`, opening a project in VS Code, etc.). A rule that fires on any of these is wrong.
6. **Look at the ancestry, not the process.** A `node` reading `~/.aws/credentials` is ambiguous; an `agent-subprocess` `node` reading it is suspicious; a `human-shell` `node` reading it (you ran a script) is fine.

If you've done all of the above and still get false positives, the rule probably doesn't belong at `high`. Drop to `medium` and let the TUI surface it without notifying.

## Performance notes

Rule evaluation is on the hot path of every filesystem event. Keep these in mind:

- **Path predicates are O(1) with prefix matching.** The engine indexes rules by path prefix so a rule on `~/.ssh/*` is only evaluated against events under `~/.ssh/`.
- **Ancestry classification is cached per-pid.** The process tree walker memoizes per-`pid` for the lifetime of the daemon's view of that pid. A burst of events from the same pid is cheap.
- **Glob is slower than `home_relative` or `starts_with`.** Prefer the cheaper predicate when both would work.
- **IoC enrichment is opportunistic.** The engine looks up the responsible package in `iocs.db` only after a rule fires — it never gates rule evaluation on IoC lookup.

## Rule lifecycle

1. **Drafted** — file in repo with `disabled: true`.
2. **Staging** — `disabled: false`, but `severity` is one level lower than intended for a 2-week soak on the corpus + at least one maintainer's workstation.
3. **Promoted** — severity raised to intended level.
4. **Deprecated** — `deprecated: true` with `replaced_by:` field. Kept for one release cycle, then removed.
